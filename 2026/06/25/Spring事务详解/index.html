Spring事务详解：@Transactional原理、传播行为、隔离级别

本文最后更新于：13 分钟前

一、Spring 事务核心概念

Spring 事务管理分为两种：

  • 编程式事务：通过 TransactionTemplate 手动控制
  • 声明式事务：通过 @Transactional 注解（推荐，最常用）

声明式事务底层基于 AOP，本质是在目标方法前后进行拦截，在方法执行前开启事务，执行后根据结果提交或回滚。

二、@Transactional 原理

实现机制

  1. 通过 @EnableTransactionManagement 启用事务管理
  2. Spring 为带有 @Transactional 的 Bean 创建代理对象
  3. 调用目标方法时，通过 TransactionInterceptor 拦截
  4. 从 PlatformTransactionManager 获取事务
  5. 执行目标方法 → 正常则提交，异常则回滚

核心接口

接口 作用
PlatformTransactionManager 事务管理器（核心）
TransactionDefinition 事务定义（传播行为、隔离级别等）
TransactionStatus 事务状态

常用事务管理器

实现类 适用场景
DataSourceTransactionManager 单数据源 JDBC/MyBatis
JpaTransactionManager JPA
JtaTransactionManager 分布式事务（多数据源）

三、事务传播行为（Propagation）

传播行为 说明
REQUIRED（默认） 支持当前事务，不存在则新建
SUPPORTS 支持当前事务，不存在则以非事务执行
MANDATORY 必须有事务，否则抛异常
REQUIRES_NEW 始终新建事务，挂起当前事务
NOT_SUPPORTED 以非事务方式执行，挂起当前事务
NEVER 以非事务方式执行，有事务则抛异常
NESTED 嵌套事务（Savepoint，部分回滚）

五、事务失效场景（高频面试）

场景 原因 解决方案
内部调用 this.method() 代理对象不生效 注入自己 Bean 或使用 AopContext
非 public 方法 CGLIB/JDK 代理限制 改为 public
异常被 catch 吞掉 未抛异常 手动抛出或设置回滚
异常类型不对 默认只回滚 RuntimeException 设置 rollbackFor = Exception.class
传播行为设置错误 SUPPORTS/NEVER 不支持事务 改为 REQUIRED
数据库不支持事务 如 MyISAM 改用 InnoDB
多数据源事务管理器配置错误 未指定正确的事务管理器 显式指定 transactionManager

七、面试题

1. @Transactional 原理？

基于 AOP 动态代理，通过 TransactionInterceptor 拦截方法，在方法前后开启/提交/回滚事务。

2. Spring 事务传播行为有哪些？

7 种：REQUIRED、SUPPORTS、MANDATORY、REQUIRES_NEW、NOT_SUPPORTED、NEVER、NESTED

3. 事务不生效的原因？

内部调用、非 public、异常被 catch、异常类型不对、传播行为不对

4. 嵌套事务 NESTED 是什么？

基于 Savepoint，内部事务回滚到 Savepoint 而非全部回滚，外层可决定是否整体提交。

5. @Transactional 加到类上和方法上有什么区别？

类级别：类中所有方法都生效；方法级别：仅该方法生效，优先级高于类级别。
