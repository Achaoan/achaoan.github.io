JVM类加载机制详解：加载、连接、初始化、双亲委派

本文最后更新于：13 分钟前

一、类加载全过程

一个类从被加载到 JVM 内存开始，到卸载出内存为止，整个生命周期包括七个阶段：

加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载

其中验证、准备、解析统称为连接。

二、加载（Loading）

做什么？

  1. 通过类的全限定名获取定义此类的二进制字节流
  2. 将字节流所代表的静态存储结构转化为方法区的运行时数据结构
  3. 在内存中生成代表这个类的 java.lang.Class 对象，作为方法区这个类的各种数据的访问入口

获取二进制字节流的方式

  • 从 ZIP/JAR 包读取（最常用）
  • 从网络获取（Applet）
  • 运行时计算生成（动态代理）
  • 由其他文件生成（JSP 转 Servlet）
  • 从数据库读取

三、验证（Verification）

确保 Class 文件的字节流中包含的信息符合 JVM 规范。

验证阶段 说明
文件格式验证 魔数 0xCAFEBABE、主次版本号、常量池常量类型
元数据验证 是否有父类、是否继承了 final 类、是否实现了所有方法
字节码验证 通过数据流分析确保语义合法
符号引用验证 是否能通过符号引用找到对应的类/方法/字段

正式为类变量（static 变量）分配内存并设置零值。

五、解析（Resolution）

将常量池内的符号引用替换为直接引用。

解析类型 示例
类/接口解析 将 CONSTANT_Class_info 解析为类指针
字段解析 将 CONSTANT_Fieldref_info 解析为字段偏移量
方法解析 将 CONSTANT_Methodref_info 解析为方法指针

符号引用：以字面量形式描述目标，与内存布局无关
直接引用：指向目标的指针、相对偏移量或句柄

六、初始化（Initialization）

执行类构造器 <clinit>() 方法的过程。

() 方法

  • 由编译器自动收集类中所有类变量赋值和静态语句块合并产生
  • 执行顺序按源文件中出现的顺序

触发场景 代码示例
new 对象、访问静态字段、调用静态方法 new Foo(), Foo.bar, Foo.baz()
反射调用 Class.forName("Foo")

七、类加载器

三层类加载器（JDK 9+）

加载器 加载路径
启动类加载器（Bootstrap） <JAVA_HOME>/lib 核心类库（rt.jar、java.base 模块）
平台类加载器（Platform，原 Extension） <JAVA_HOME>/lib/ext 或 java.ext.dirs
应用类加载器（Application / System） classpath 或 -cp 指定的类

双亲委派模型

为什么需要双亲委派？

  1. 安全：保证核心 API 不被篡改（不会出现多个版本的 java.lang.Object）
  2. 避免重复加载：父加载器已加载的类不会被子加载器重复加载

如何打破双亲委派？

  • 重写 loadClass() 方法（而不是 findClass()）
  • 典型场景：Tomcat 为每个 WebApp 提供独立类加载器；SPI 机制（JDBC 驱动加载）

八、常见面试题

1. 类加载和”双亲委派”的执行过程？

见上。先委派父加载器，父无法加载才自加载。

2. 一个类是如何被加载到 JVM 的？

经过加载→验证→准备→解析→初始化 五个阶段。

3. 能否自己写一个 java.lang.String 并替换？

不能。双亲委派模型保证核心库由启动类加载器加载，自定义的 java.lang.String 不会被加载。

4. 两个相同类名但由不同类加载器加载的类是否相等？

不相等。instanceof 结果为 false。

5. 什么是 SPI？如何打破双亲委派？

SPI（Service Provider Interface）如 JDBC，由启动类加载器加载接口但需要调用应用类加载器的实现类。使用线程上下文类加载器（Thread Context ClassLoader）打破双亲委派。
