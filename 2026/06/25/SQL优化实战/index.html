SQL优化实战：Explain执行计划、索引优化、慢查询

本文最后更新于：1 小时前

一、SQL 执行计划（EXPLAIN）

使用 EXPLAIN 关键字可以模拟优化器执行 SQL 查询，从而知道 MySQL 是如何处理你的 SQL 语句的。

EXPLAIN 输出列

列名 说明
id 查询编号，id 越大越先执行
select_type 查询类型（SIMPLE、PRIMARY、SUBQUERY、DERIVED、UNION）
table 正在访问的表
partitions 匹配的分区
type 访问类型（重点关注）
possible_keys 可能使用的索引
key 实际使用的索引
key_len 索引长度
ref 与索引比较的列或常量
rows 预计扫描的行数（越少越好）
filtered 按表条件过滤的行百分比
Extra 额外信息（重点关注）

type 访问类型（性能从好到差）

type 说明 场景
system 表只有一行（系统表） 极少见
const 主键或唯一索引等值匹配 WHERE id = 1
eq_ref 联表查询，被驱动表使用主键/唯一索引 JOIN ... ON a.id = b.user_id
ref 普通索引等值匹配 WHERE name = '张三'（非唯一索引）
range 索引范围扫描 WHERE age > 18、BETWEEN、IN
index 全索引扫描 遍历索引树
ALL 全表扫描 需要优化的信号

Extra 关键信息

Extra 说明
Using index 覆盖索引，不需要回表
Using where 在存储引擎层过滤后还需在 Server 层过滤
Using index condition 索引下推（ICP），减少回表
Using filesort 文件排序，需要优化
Using temporary 使用临时表，需要优化
Using join buffer 联表未用索引，使用了 Join Buffer

二、索引优化策略

最左前缀法则

联合索引 (a, b, c) 的生效规则：

原则：查询从索引的最左列开始，不能跳过中间列。

索引失效场景

场景 错误示例 正确写法
隐式类型转换 WHERE phone = 13800001111 WHERE phone = '13800001111'
索引列参与计算 WHERE age + 1 = 20 WHERE age = 19
索引列使用函数 WHERE LEFT(name,1) = '张' 考虑冗余字段或全文索引
LIKE 以通配符开头 WHERE name LIKE '%明' WHERE name LIKE '张%'
OR 存在非索引列 WHERE id = 1 OR age = 20 拆分为 UNION
!= / <> 不等条件 WHERE status != 1 视数据分布而定
NOT IN WHERE id NOT IN (1,2,3) 改用 NOT EXISTS

覆盖索引

定义：查询的所有列都在索引中，无需回表查询。

索引下推（ICP）

MySQL 5.6+ 优化：在索引遍历过程中，对索引包含的列先做过滤，减少回表次数。

七、面试题

1. Explain 中 type 各值的含义和区别？

system > const > eq_ref > ref > range > index > ALL，ALL 需要优化。

2. 索引失效的场景有哪些？

隐式类型转换、索引列计算/函数、LIKE 以 % 开头、OR 包含非索引列、NOT IN 等。

3. 什么是覆盖索引和索引下推？

覆盖索引：查询列都在索引中，无需回表。索引下推（ICP）：在索引层先过滤再回表。

4. 慢查询如何定位和优化？

开启慢查询日志 → mysqldumpslow 分析 → EXPLAIN 分析执行计划 → 针对性加索引或改写 SQL。

5. 深分页为什么慢？如何优化？

LIMIT 100000, 20 需要扫描 100020 行后丢弃前 100000 行。优化：游标分页或延迟关联。

6. count(*), count(1), count(列) 的区别？

COUNT(*) 和 COUNT(1) 统计行数，MySQL 已优化；COUNT(列) 统计非 NULL 行数。
