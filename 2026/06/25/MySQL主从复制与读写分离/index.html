二、复制原理（三步骤）

三个线程

线程 位置 作用
Binlog Dump Thread 主库 读取 binlog 发送给从库
IO Thread 从库 接收主库 binlog 写入 relay log
SQL Thread 从库 读取 relay log 回放 SQL

两个日志

日志 说明
binlog（二进制日志） 记录主库的所有数据变更
relay log（中继日志） 从库暂存从主库拉取的 binlog

三、Binlog 详解

三种格式

格式 说明 优点 缺点
STATEMENT 记录 SQL 语句 日志量小 非确定性函数（NOW()）可能导致主从不一致
ROW（推荐） 记录每行数据变更 一致性最好 日志量大
MIXED 混合模式，默认 STATEMENT，非确定性时用 ROW 折中方案 复杂场景仍可能不一致

Binlog 写入机制

sync_binlog 参数：

  • 0：由 OS 决定刷盘时机（性能最好，丢数据风险最大）
  • 1：每次提交事务都刷盘（最安全）
  • N：每 N 次事务提交刷盘一次

读写分离的挑战

挑战 原因 解决方案
主从延迟 从库回放需要时间 强制读主、延迟监控、分库设计
事务一致性 写后立即读可能读不到 写后读走主库

六、主从延迟问题

延迟原因

监控延迟

解决方案

方案 说明
强制读主 对一致性敏感的场景直接读主库
延迟监控 设置阈值，超过则自动切换读主
缓存先写 写入后更新 Redis，从库延迟时从缓存读
多线程回放 slave_parallel_workers = 4 + slave_parallel_type = LOGICAL_CLOCK
避免大事务 大批量操作分批处理

八、面试题

1. MySQL 主从复制的原理是什么？

主库数据变更写入 binlog → 从库 IO 线程拉取写入 relay log → 从库 SQL 线程回放 relay log。

2. binlog 的三种格式区别？

STATEMENT（记录 SQL）、ROW（记录行变更，推荐）、MIXED（混合模式）。

3. 主从延迟如何解决？

强制读主、多线程回放、缓存辅助、避免大事务、提升从库配置。

4. 半同步复制和异步复制的区别？

异步：主库提交即返回，可能丢数据；半同步：需至少一个从库确认，基本不丢数据。

5. 如何实现读写分离？

应用层（Spring AbstractRoutingDataSource）或中间件（ShardingSphere、ProxySQL）。

6. GTID 复制有什么好处？

主从切换时无需指定日志位置，自动基于 GTID 找同步点，简化运维。
