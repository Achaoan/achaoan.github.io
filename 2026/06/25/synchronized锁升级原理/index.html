synchronized锁升级原理：偏向锁、轻量级锁、重量级锁

本文最后更新于：20 分钟前

一、概述

JDK 6 之前，synchronized 只有重量级锁，依赖 Mutex Lock，线程阻塞/唤醒需要在用户态与内核态之间切换，性能开销大。

JDK 6 引入锁升级机制，根据竞争激烈程度动态选择锁策略：

无锁 → 偏向锁 → 轻量级锁 → 重量级锁

升级是单向不可逆的。

二、Mark Word（对象头）

每个 Java 对象的对象头（Mark Word）中存储着锁状态信息：

锁状态 存储内容 锁标志位
无锁 对象哈希码、分代年龄 01
偏向锁 线程 ID、Epoch、分代年龄 01（偏向位=1）
轻量级锁 指向线程栈锁记录的指针 00
重量级锁 指向 Monitor 对象的指针 10

三、偏向锁（Biased Locking）

设计目的

假设同步块绝大多数时候只有一个线程访问，消除同一线程重复获取锁的开销。

实现原理

  • 第一次获取：CAS 将当前线程 ID 写入 Mark Word，锁标志位设为 101
  • 后续重入：只需检查线程 ID 是否是自己，直接执行，无需同步操作

撤销条件

当其他线程尝试竞争该锁时：

  1. 到达全局安全点（SafePoint），暂停持有偏向锁的线程
  2. 检查该线程是否仍在执行同步块：
    • 已退出 → 恢复为无锁状态
    • 仍在执行 → 升级为轻量级锁

注意事项

  • JDK 15+ 已默认禁用偏向锁
  • 调用 hashCode() 或 wait() 直接导致偏向锁升级
  • 偏向锁默认有 4 秒延迟启动（-XX:BiasedLockingStartupDelay=0 可关闭）

四、轻量级锁（Lightweight Locking）

设计目的

多个线程交替访问同步块（无实际竞争），避免重量级锁的内核态切换。

加锁过程

  1. 线程在当前栈帧创建锁记录（Lock Record）
  2. 复制对象头 Mark Word 到锁记录（Displaced Mark Word）
  3. CAS 操作尝试将对象头的 Mark Word 替换为指向锁记录的指针：
    • 成功 → 获取锁，标志位 00
    • 失败 → 检查是否自己线程的锁记录：
      • 是 → 锁重入
      • 否 → 自旋尝试获取

自旋锁 & 自适应自旋

  • JDK 1.6+：自适应自旋——JVM 根据上次自旋成功率动态调整
    • 上次成功 → 多自旋几次
    • 上次失败 → 减少或跳过自旋

升级为重量级锁的条件

  • 自旋超过阈值仍未获取锁
  • 三个及以上线程同时竞争

五、重量级锁（Heavyweight Locking）

ObjectMonitor

依赖 ObjectMonitor 对象（C++ 实现），包含：

组件 作用
_owner 持有锁的线程
_cxq 竞争队列
_EntryList 候选者队列
_WaitSet 调用 wait() 的线程
_recursions 重入次数

加锁流程

  1. 尝试获取 _owner，成功则进入
  2. 失败 → 先尝试自旋
  3. 自旋失败 → 线程阻塞（进入内核态），放入等待队列

七、JVM 锁优化

优化 说明
锁粗化 多个连续的加锁/解锁合并为一个
锁消除 对不可能有竞争的锁直接消除
适应性自旋 根据历史成功率调整自旋次数
批量重偏向 同一类偏向锁撤销达 20 次时触发
批量撤销 同一类偏向锁撤销达 40 次时禁用该类偏向锁

八、实践建议

  1. 低竞争场景：JVM 锁升级足够优秀，无需额外优化
  2. 高并发场景：
    • 可禁用偏向锁：-XX:-UseBiasedLocking
    • 考虑 ReentrantLock、StampedLock
  3. 缩小锁粒度：减小同步代码块范围
  4. 监控：使用 JOL（Java Object Layout）查看对象头信息
