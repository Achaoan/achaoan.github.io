String深入详解：常量池、不可变性、拼接原理与intern

本文最后更新于：1 小时前

一、String 的不可变性

所有的”修改”操作都不会改变原字符串，而是创建并返回一个新字符串对象。

源码层面的设计

  • final 修饰类：防止被继承后篡改行为
  • private final char[] value：数组引用不可变，且外部不可直接访问
  • 不提供 setter 方法

为什么设计为不可变？

  1. 字符串常量池的需要：相同字面量的 String 可安全共享引用
  2. HashMap 的 key 安全：若可变则 hashCode 会变化，导致找不到 value
  3. 线程安全：不可变对象天然线程安全
  4. 类加载机制安全：类加载中大量使用字符串
  5. 缓存 hashCode：不可变后 hashCode 可直接缓存

二、字符串常量池（String Pool）

字符串常量池本质是一个 HashTable，用于实现字符串的复用。

JDK 版本 常量池位置 说明
JDK 6 及之前 方法区（永久代） 与堆分离
JDK 7 堆中 从永久代移入堆
JDK 8+ 堆中 永久代被元空间取代

创建对象的个数

String s = new String("abc") — 创建了几个对象？

结论：1 个或 2 个

  • 若常量池已有 “abc”：只在堆中创建 1 个对象
  • 若常量池没有 “abc”：先在常量池创建 1 个，再在堆中创建 1 个，共 2 个

new String("a") + new String("b") — 创建了几个对象？

结论：5 个对象

  • 常量池中：”a”、”b”（2 个）
  • 堆中：”a” 对象、”b” 对象、”ab” 拼接结果对象（3 个）
  • 常量池中没有 “ab”（拼接结果不会自动入池）

三、字符串拼接原理

编译期优化：常量折叠

运行时优化：StringBuilder

性能陷阱

场景 结果位置 原理
常量 + 常量 常量池 编译期优化
常量 + 变量 堆中 StringBuilder.append
final 常量 + 常量 常量池 编译期折叠
拼接后 intern() 常量池 主动入池

四、intern() 方法

  • 常量池已存在相同字符串 → 返回常量池中的引用
  • 常量池不存在 → 将字符串放入常量池，返回引用

JDK 6 vs JDK 7+ 区别

JDK 版本 常量池不存在时的行为
JDK 6 在常量池中深拷贝创建新对象
JDK 7+ 在常量池中保存堆中对象的引用
