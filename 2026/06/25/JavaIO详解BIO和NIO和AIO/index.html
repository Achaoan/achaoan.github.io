Java IO详解：BIO、NIO、AIO、零拷贝

本文最后更新于：2 小时前

一、BIO / NIO / AIO 核心对比

维度 BIO NIO AIO
全称 Blocking I/O（同步阻塞） Non-blocking I/O（同步非阻塞） Asynchronous I/O（异步非阻塞）
线程模型 一个连接一个线程 一个请求一个线程（多路复用） 回调驱动
阻塞阶段 等待数据+拷贝均阻塞 等待数据非阻塞，拷贝阻塞 两阶段均非阻塞
吞吐量 低 高 极高
编程复杂度 低 高 中
适用场景 低并发、短连接 高并发、长连接 大文件、IO密集型

二、BIO（Blocking I/O）

原理

同步阻塞，accept()、read() 方法会阻塞线程直到完成。

痛点

  • 线程数随连接数线性增长（C10K 问题）
  • 线程上下文切换开销大

三、NIO（Non-blocking I/O）

核心三件套

组件 说明
Channel（通道） 双向，可读可写，替代 BIO 的单向流
Buffer（缓冲区） 数据读写载体，支持 DirectByteBuffer
Selector（选择器） 多路复用器，单线程管理多个 Channel

Buffer 核心方法

  • flip()：切换为读模式
  • clear()：清除缓冲区
  • rewind()：重绕，position 置 0

Selector 在 Linux 的实现

基于 epoll，三个 native 方法封装：

  • epoll_create → epoll_ctl → epoll_wait

NIO 服务端流程

同步与异步、阻塞与非阻塞

  • 同步：IO 操作完成由线程自身主动等待
  • 异步：IO 操作完成后由系统通知线程
  • 阻塞：线程发起 IO 后必须等待操作完成
  • 非阻塞：线程发起 IO 后无需等待

四、AIO（Asynchronous I/O）

采用 Proactor 模式，由 OS 内核完成 IO 后主动回调。

局限性

  • Linux 平台 AIO 实现不完善（底层仍用 epoll 模拟）
  • 业界主流仍以 NIO 为主（如 Netty）

五、零拷贝（Zero-Copy）

实现方式

1. DirectByteBuffer（堆外内存）

减少 JVM 堆内存到直接内存的拷贝。

2. FileChannel.transferTo()
底层调用 sendfile 系统调用，数据直接从文件内核缓冲区发送到 Socket。

3. MappedByteBuffer（内存映射文件）

4. Netty 中的零拷贝

  • 接收和发送默认使用 Direct Buffer
  • CompositeByteBuf：合并多个 ByteBuf 避免内存拷贝
  • FileRegion：包装 FileChannel.transferTo

六、select/poll/epoll 区别

机制 复杂度 特点
select O(n) 线性扫描 fd 集合，最大 1024
poll O(n) 链表突破数量限制，仍线性扫描
epoll O(1) 回调机制，支持边缘触发（ET）

七、技术选型

场景 推荐模型
低并发、短连接 BIO
高并发网络通信 NIO（Netty / Tomcat NIO）
大文件传输优化 零拷贝（transferTo + DirectBuffer）
Java基础
Java 面试 IO NIO 网络编程
