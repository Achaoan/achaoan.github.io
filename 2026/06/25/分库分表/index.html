分库分表详解：垂直拆分、水平拆分、ShardingSphere

本文最后更新于：16 分钟前

一、为什么需要分库分表？

单库单表的瓶颈

瓶颈类型 说明
数据量过大 单表数据量达到千万/亿级，索引变深，B+ 树层级增加，IO 次数增多
QPS 过高 单库连接数有限，CPU/内存/IO 成为瓶颈
写入压力 单库写入无法水平扩展，主从复制延迟

分库分表解决的问题

分库分表的时机

  • 单表数据量超过 500万~1000万（视字段多少和访问模式而定）
  • 单库 QPS 持续超过数据库实例上限
  • 写入量超过单个主库的处理能力
  • 磁盘 IO 成为瓶颈

分片策略

策略 说明 公式 适用场景
取模/哈希 根据分片键哈希值取模 id % N 数据分布均匀
范围分片 按时间或 ID 范围划分 id ∈ [1,1000万] → shard_0 范围查询友好
一致性哈希 最小化扩缩容影响 hash(key) 落在环上 需要弹性伸缩
时间分片 按月/日分表 order_202601, order_202602 日志类数据

分片键选择

  • 订单场景：按用户 ID 分片（用户维度的查询最频繁）
  • 时序数据：按时间范围分片
  • 需覆盖大部分查询：尽量让分片键出现在 WHERE 条件中

跨分片问题

问题 解决方案
跨分片查询 结果集归并（ShardingSphere 自动处理）
跨分片排序 各分片排序后归并
跨分片分页 需要取所有分片前 N 条再归并（深分页性能差）
分布式事务 Seata、TCC、消息最终一致性
全局主键 雪花算法（Snowflake）、UUID、号段模式

七、面试题

1. 分库分表后带来的问题？

  • 跨分片查询（需要归并）
  • 分布式事务
  • 全局主键生成
  • 分页、排序复杂度增加
  • 数据迁移困难

2. 分片键如何选择？

选择查询频率最高的字段，如用户维度的系统用 user_id，保证大部分查询带分片键。

3. 取模分片扩容怎么办？

  • 一致性哈希：最小化迁移数据
  • 翻倍扩容：从 N 扩到 2N，每个分片一半数据迁移
  • Range + Mod：先按范围再取模

4. ShardingSphere 和 MyCAT 的区别？

ShardingSphere 更活跃，支持 JDBC 和 Proxy 两种模式；MyCAT 基于 Proxy 模式，功能相对较少。

5. 分布式事务如何实现？

Seata AT/TCC 模式、MQ 最终一致性、本地消息表，根据一致性要求选择。

6. 跨分片查询如何归并排序？

各分片返回前 N 条 → 内存中堆排序归并 → 返回最终结果（数据量大时性能差）。
