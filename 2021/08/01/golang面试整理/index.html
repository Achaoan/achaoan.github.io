golang面试整理

本文最后更新于：3 小时前

1. 项目中用到的锁

Go 中锁分为两种：互斥锁 和 读写锁。

互斥锁

并发程序对公共资源访问的限制最常见的就是使用互斥锁的方式。在 Go 中，sync.Mutex 提供了互斥锁的实现。

读写锁

读写锁是对读写操作进行加锁。多个读操作之间不互斥，提高对共享资源的访问效率。

Go 中读写锁由 sync.RWMutex 提供：

  • Lock() / Unlock()：写锁定/解锁，锁定后其他 goroutine 不能读或写
  • RLock() / RUnlock()：读锁定/解锁，锁定后其他 goroutine 不能写，但可以读

2. 介绍一下线程安全的共享内存方式

Go 提倡”通过通信来共享内存，而不是通过共享内存来通信”。推荐使用 channel 进行 goroutine 间通信。若必须使用共享内存，则用 sync.Mutex 或 sync.RWMutex 保护。

3. 介绍一下 goroutine

goroutine 是 Go 语言轻量级线程（协程），由 Go runtime 管理。栈初始仅 2-4KB，可动态伸缩。通过 go 关键字创建，调度由 GMP 模型实现（Goroutine-Machine-Processor）。

4. goroutine 的自旋占用资源如何解决，GMP 模型？

GMP 模型：

  • G：Goroutine，包含栈、状态、任务函数
  • M：Machine（线程），与操作系统线程一一对应
  • P：Processor（逻辑处理器），本地 goroutine 队列（默认数量 = GOMAXPROCS）

调度流程：

  1. 全局 G 队列 → 分配到 P 的本地队列
  2. M 从 P 的本地队列获取 G 执行
  3. P 本地队列空时窃取其他 P 的 G（work stealing）
  4. G 阻塞时 M 与 P 解绑，P 寻找新的 M

自旋问题： M 没有 G 可执行时会在用户态自旋（占用 CPU），而不是休眠。解决：通过 runtime.Gosched() 让出 CPU，或调整 GOMAXPROCS 控制 P 的数量。

5. 介绍 Linux 系统信号

  • SIGTERM（15）：终止信号，可捕获处理
  • SIGKILL（9）：强制终止，不可捕获
  • SIGINT（2）：Ctrl+C 中断
  • SIGQUIT（3）：Ctrl+\ 退出 + core dump
  • Go 通过 signal.Notify() 捕获信号做优雅退出

6. goroutine 抢占时机，GC 栈扫描

  • 1.14 之前：协作式抢占（函数调用时检查抢占标记）
  • 1.14+：基于信号的异步抢占（发送 SIGURG 信号）
  • GC 栈扫描：STW 时扫描所有 P 的 goroutine 栈，找到根对象

7. GC 触发时机

  • 主动触发：调用 runtime.GC()
  • 内存分配阈值：堆内存达到上次 GC 后存活堆内存的一定比例（默认 100%）
  • 定时触发：超过 2 分钟未 GC 则强制触发

8. 是否了解其他 GC 机制？

  • 引用计数（Python/PHP）：每个对象维护引用计数，循环引用无法回收
  • 标记-清除（Java G1）：分代收集
  • 三色标记法（Go 1.5+）：并发标记，黑色/灰色/白色

9. 内存管理方式

Go 使用 mcache→mcentral→mheap 三级内存管理：

  • mcache：每个 P 的本地缓存（无锁分配）
  • mcentral：全局中心缓存，按大小分类
  • mheap：堆，向操作系统申请内存

大小分类：微对象 < 16B（Tiny allocator），小对象 < 32KB，大对象 ≥ 32KB。

10. Channel 分配在堆上还是在栈上？哪些对象分配在堆上？

Channel 分配在堆上（因为需要跨 goroutine 共享）。Go 编译器通过逃逸分析决定分配位置：变量地址被外部引用（逃逸）时分配在堆上，否则在栈上。

11. 代码效率分析，考虑局部性原理

  • 时间局部性：循环中的重复变量应尽量复用
  • 空间局部性：数组遍历比链表遍历快（连续内存访问缓存友好）
  • Go 中切片遍历比 map 遍历更有空间局部性

12. 多核 CPU 下，cache 如何保持一致，不冲突？

MESI 协议（缓存一致性协议）：

  • M（Modified）：缓存行已修改
  • E（Exclusive）：独占
  • S（Shared）：共享
  • I（Invalid）：无效
    多核通过嗅探机制（snooping）监听总线，使过期缓存行失效。

13. uint 类型溢出

uint 在 32 位系统上是 32 位（0 ~ 4294967295），64 位系统是 64 位。溢出时回绕（wraparound），如 math.MaxUint64 + 1 = 0。

14. 聊聊 rune 类型

rune 是 int32 的别名，用于表示 Unicode 码点（code point）。Go 中字符串是字节序列，使用 []rune(str) 可正确处理 Unicode 字符。

15. 介绍一下 channel，有缓冲和无缓冲的区别

  • 无缓冲 make(chan int)：同步，发送和接收必须同时准备好
  • 有缓冲 make(chan int, 10)：异步，缓冲区满时发送阻塞

特性：

  • 关闭后：读取已发送值，再读返回零值
  • v, ok := <-ch：通过 ok 判断 channel 是否关闭
  • nil channel 读写都会阻塞

16. channel 是否线程安全？

是的。channel 本身通过内置锁机制保证线程安全，多个 goroutine 同时读写不会出现 data race。

17. 介绍一下 Mutex 的实现，是悲观锁还是乐观锁？

Mutex 是悲观锁，基于 CAS + 自旋 + 信号量实现。Go 1.9+ 的 Mutex 分为正常模式和饥饿模式：

  • 正常模式：自旋等待 + 队列 FIFO
  • 饥饿模式：避免 goroutine 长时间获取不到锁

18. Mutex 几种模式？

  • 正常模式：goroutine 自旋几次后进入等待队列，先入先出但可能被新来的 goroutine 插队
  • 饥饿模式：等待队列中的 goroutine 等待时间超过 1ms 触发，直接传递锁，消除插队

19. Mutex 可以做自旋锁？

可以，但 Go 没有直接提供自旋锁 API。可通过 sync.Mutex 配合自旋实现，或使用 sync/atomic 操作自行实现自旋锁。

20. 介绍一下 RWMutex

读写锁，适合读多写少的场景：

  • 多读可以同时进行（RLock）
  • 写独占（Lock）
  • 写优先级：有写等待时，新读请求会阻塞

21. 介绍一下大对象和小对象，为什么小对象多了会造成 GC 压力？

  • 大对象：≥ 32KB，分配和回收成本高（直接 GC 扫描）
  • 小对象：< 32KB，通过 mcache 分配。大量小对象导致 GC 标记阶段遍历时间长，STW 压力大

22. 介绍项目中遇到的 OOP 情况

Go 通过 struct + 方法实现 OOP，通过组合（embedding）实现复用，通过接口实现多态。

23. 介绍项目中遇到的坑

常见陷阱：

  • 循环变量捕获（goroutine 中使用循环变量）
  • map 并发读写 panic（需加锁）
  • interface 比较（nil interface 和 nil 指针的区别）

24. 如何指定指令执行的顺序？

使用 sync/atomic 的原子操作保证可见性和顺序，或使用 sync.Mutex 建立 happens-before 关系。

25. 什么是写屏障、混合写屏障，如何实现？

GC 三色标记中的并发保护机制：

  • 写屏障：对象引用变化时触发，防止漏标
  • 混合写屏障（Go 1.8+）：结合插入写屏障和删除写屏障，减少 STW 时间

原理：GC 开始时扫描所有栈，后续栈对象变更通过写屏障记录。

26. GC 的 STW 是怎么回事？

Stop The World，GC 时暂停所有 goroutine 以保证一致性视图。Go 通过并发 GC 大幅减少 STW 时间（1.8+ 通常在 100μs-1ms 级）。

27. 协程之间是怎么调度的？

通过 GMP 模型：M 从 P 取 G 执行，阻塞时 M 和 P 解绑，新建或唤醒 M 执行其他 G。

28. 简单聊聊内存逃逸

变量在函数返回后仍被使用时，编译器将其分配到堆上。常见逃逸场景：

  • 返回局部变量的指针
  • 大对象（编译器判定）
  • 闭包引用的变量
  • interface{} 类型变量（动态类型）

31. http 包的内存泄漏

常见场景：

  • defer resp.Body.Close() 未调用
  • 未读取完整响应体直接关闭
  • 长连接未设置超时

32. Goroutine 调度策略

  • work stealing：P 本地队列空时窃取其他 P 的 goroutine
  • hand off：G 阻塞时 M-P 解绑，P 绑定其他 M
  • 并行：多核并行执行 GOMAXPROCS 个 goroutine

33. 对已经关闭的 chan 进行读写，会怎么样？为什么？

  • 读：返回已发送的值和零值（v, ok := <-ch 中 ok=false）
  • 写：panic（close of closed channel）
  • 关闭：double close 也会 panic

34. 实现阻塞读的并发安全 Map

使用 sync.RWMutex 保护 map 读写，或者使用 sync.Map（Go 1.9+）。

35. 什么是 goroutine leak？

goroutine 启动后一直阻塞无法退出，导致内存泄漏。常见场景：

  • channel 读写不匹配导致阻塞
  • select 中所有 case 不满足
  • 死循环缺少退出条件

36. data race 问题怎么解决？能不能不加锁解决这个问题？

  • 加锁（Mutex/RWMutex）
  • 使用 channel 通信
  • 使用 sync/atomic 原子操作
  • 避免共享（每个 goroutine 独立数据）

不加锁解决：使用 channel 替代共享内存通信。

37. epoll 原理

Linux 多路复用 IO 模型：

  1. epoll_create 创建 epoll 实例
  2. epoll_ctl 注册文件描述符监听事件
  3. epoll_wait 阻塞等待事件发生

使用红黑树管理 fd，就绪链表返回事件，O(1) 复杂度。Go 的 netpoller 基于 epoll。

38. etcd 怎么实现分布式锁？

基于 lease（租约） + watch（监听）：

  • grant 创建 lease
  • put 带 lease 写入 key
  • txn 事务判断 key 不存在则创建（原子性）
  • 通过 watch 监听锁释放

39. 滑动窗口的概念以及应用？

网络流量控制算法。发送方维护窗口大小，窗口内可发送无需等待 ACK，窗口滑动后释放新空间。
应用：TCP 流量控制、限流、数据流统计。

40. grpc 内部原理是什么？

基于 HTTP/2 的 RPC 框架：

  • Protocol Buffers 序列化
  • HTTP/2 多路复用（单一 TCP 连接支持多个流）
  • 支持四种通信模式：一元、服务端流、客户端流、双向流

41. HTTP/2 的特点是什么，与 HTTP/1.1 的对比

  • 多路复用：单一连接并发请求
  • 头部压缩：HPACK 压缩算法
  • 服务器推送：服务端主动推送资源
  • 二进制分帧：取代 HTTP/1.1 的文本格式
  • 优先级：可设置请求优先级

42. time.Now 有几次系统调用？如何优化？

通常一次系统调用（vdso 优化后 0 次直接系统调用）。vdso 将内核部分代码映射到用户空间，避免上下文切换。

43. 空 struct{} 是否使用过？会在什么情况下使用？

空 struct 不占用内存。使用场景：

  • 实现 Set（map[string]struct{}）
  • 信号 channel（chan struct{}）
  • 只有方法没有属性的对象

44. 聊聊 runtime

Go runtime 提供：调度器（GMP）、内存分配器（mcache/mcentral/mheap）、垃圾回收器（并发三色标记）、网络轮询器（netpoller）、信号处理、反射、panic 恢复等。

45. 介绍下你平时都是怎么调试 bug 以及性能问题的？

  • 调试：pprof（cpu/heap/goroutine/mutex）、trace、delve 断点、log 日志
  • 性能：pprof 火焰图、benchmark 基准测试、pprof 分析内存和 CPU
  • 排查：GODEBUG 环境变量、go vet 静态分析、race detector 竞态检测

46. 通过通信来共享内存，而不是通过共享内存而通信，怎么理解？

Go 哲学：使用 channel 在 goroutine 间传递数据的所有权，而不是用锁保护共享变量。channel 保证了数据在同一时刻只被一个 goroutine 访问，降低心智负担和出错概率。

47. chan 比 mutex 更轻么？还有更轻量的方法么？

channel 比 mutex 更重（涉及 goroutine 调度、调度器唤醒）。更轻量的方法：sync/atomic 原子操作（无锁）、sync.Pool 对象池。

48. 什么时候用 chan 不如 mutex 效率高？

  • 简单共享变量保护（计数器、状态标志）→ 原子操作或 Mutex 更快
  • 大量高频短临界区操作 → Mutex 的开销小于 channel 调度
  • 读多写少 → RWMutex 性能远优于 channel
