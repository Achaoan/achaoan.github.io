Redis基础及相关问题

本文最后更新于：3 小时前

一、Redis 简介

Redis（Remote Dictionary Server）是一个基于内存的键值存储系统，常用作数据库、缓存、消息中间件。主要特点：

  • 高性能：读 10万+/s，写 8万+/s
  • 丰富的数据结构：String、Hash、List、Set、ZSet 等
  • 持久化：RDB（快照）和 AOF（追加文件）
  • 高可用：主从复制、哨兵模式、Redis Cluster

Redis 为什么快？

  1. 纯内存操作：数据存储在内存中，读写速度极快
  2. 单线程模型：避免锁竞争和上下文切换的开销
  3. IO 多路复用：基于 epoll 的非阻塞 IO 模型
  4. 高效数据结构：SDS、跳表、压缩列表等底层数据结构优化

二、五种基础数据类型

类型 底层编码 应用场景
String SDS（简单动态字符串）、int、embstr、raw 缓存、计数器（INCR）、分布式锁（SETNX）、Session
Hash ziplist（小数据量）、hashtable 存储对象（用户信息、商品属性）
List quicklist（ziplist + linkedlist） 消息队列、最新消息列表、时间轴
Set intset（整数集合）、hashtable 去重、共同好友（SINTER）、标签系统
ZSet skiplist + dict（哈希表） 排行榜（ZREVRANGE）、延迟队列

三种高级数据结构

  • Bitmap：基于 String 实现，用户签到、日活统计
  • HyperLogLog：UV 统计（PFADD，误差 0.81%）
  • GEO：基于 ZSet，地理位置查询（GEORADIUS）

三、持久化机制

对比维度 RDB AOF
原理 定时生成数据快照（dump.rdb） 记录每次写操作命令到日志文件
文件大小 小（二进制压缩） 大（可 rewrite 压缩）
恢复速度 快 慢
数据安全性 可能丢失最后一次快照后的数据 可配置每秒同步 / 每次写同步
触发方式 SAVE（阻塞）/ BGSAVE（fork 子进程） 写操作实时追加 + 后台重写

混合持久化（Redis 4.0+）：RDB 全量快照 + AOF 增量日志，兼顾启动速度和数据安全性。

四、常见面试题

1. 缓存穿透、缓存击穿、缓存雪崩的区别及解决方案？

问题 现象 解决方案
缓存穿透 查询不存在的数据，缓存和数据库都没有 布隆过滤器、缓存空值（短过期时间）
缓存击穿 热点 key 过期，高并发访问数据库 互斥锁、逻辑过期时间、永不过期
缓存雪崩 大量 key 同时过期，请求打到数据库 随机过期时间、多级缓存、限流降级

2. Redis 内存淘汰策略有哪些？

  • noeviction：不淘汰，直接返回错误（默认）
  • allkeys-lru：所有 key 中淘汰最近最少使用的
  • volatile-lru：设置了过期时间的 key 中淘汰最近最少使用的
  • allkeys-random：所有 key 中随机淘汰
  • volatile-random：设置了过期时间的 key 中随机淘汰
  • volatile-ttl：淘汰过期时间最近的 key

3. Redis 的过期删除策略？

  • 定时删除：设置过期时间的同时创建一个定时器（CPU 不友好）
  • 惰性删除：访问 key 时检查是否过期（内存不友好）
  • 定期删除：每隔一段时间随机抽取部分 key 检查并删除（折中方案）

5. Redis 与 Memcached 的区别？

  • Redis 支持多种数据结构，Memcached 仅支持简单 KV
  • Redis 支持持久化，Memcached 不支持
  • Redis 支持原生集群，Memcached 不支持
  • Redis 单线程，Memcached 多线程

6. Redis 主从复制的原理？

  1. 从节点发送 psync 命令请求同步
  2. 主节点执行 BGSAVE 生成 RDB 快照，同时缓冲写命令
  3. 主节点将 RDB 文件发送给从节点
  4. 从节点加载 RDB，然后主节点将缓冲的写命令发送给从节点（增量同步）
  5. 后续主从之间通过长连接持续同步写操作

7. 哨兵（Sentinel）模式的工作流程？

哨兵用于监控主节点状态，实现自动故障转移：

  1. 监控：哨兵节点定期向所有节点发送心跳
  2. 选举：主节点宕机后，哨兵集群选举 leader
  3. 故障转移：从从节点中选举一个新主节点
  4. 通知：更新客户端配置，指向新主节点

8. Redis Cluster 的数据分片原理？

  • 使用 16384 个哈希槽（hash slot）
  • 通过 CRC16(key) % 16384 计算 key 属于哪个槽
  • 每个节点负责一部分哈希槽
  • 添加或删除节点时，将部分哈希槽迁移到其他节点

9. 如何保证缓存与数据库的数据一致性？

  • 先更新数据库，再删除缓存（推荐）
  • 使用消息队列异步重试
  • 设置合理的过期时间作为兜底
  • 延时双删：更新数据库后延迟一段时间再删除缓存
