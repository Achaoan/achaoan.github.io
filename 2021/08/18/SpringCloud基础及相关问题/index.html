SpringCloud基础及相关问题

本文最后更新于：2 小时前

一、什么是微服务？

微服务架构是一种将单一应用程序划分为一组小型服务的设计方法，每个服务运行在自己的进程中，通过轻量级通信机制（通常是 HTTP/REST）相互协作。

特点：

  • 单一职责：每个服务只负责一个业务功能
  • 独立部署：可独立开发、测试、部署、扩展
  • 轻量级通信：通过 RESTful API 或 RPC 通信
  • 去中心化管理：每个服务可使用不同的技术栈

二、Spring Boot 和 Spring Cloud 的关系

  • Spring Boot：关注单个微服务的快速开发，提供自动配置、内嵌服务器等
  • Spring Cloud：关注全局微服务治理，提供服务注册发现、配置中心、网关等
  • Spring Boot 是 Spring Cloud 的基础，Spring Cloud 在 Spring Boot 基础上构建

三、核心组件详解

1. 服务注册与发现（Eureka / Nacos）

Eureka：

  • Eureka Server 作为注册中心，提供服务注册和发现能力
  • 自我保护机制：15 分钟内 85% 节点心跳丢失时触发，不剔除服务
  • CAP 理论中保证 AP（可用性 + 分区容错性）

Eureka 与 ZooKeeper 对比：

  • Eureka：AP，最终一致性
  • ZK：CP，强一致性（Leader 宕机时不可用）

2. 负载均衡（Ribbon / LoadBalancer）

Ribbon 负载均衡策略：

  • 轮询（RoundRobinRule）：默认策略
  • 随机（RandomRule）
  • 响应时间加权（WeightedResponseTimeRule）
  • 最小并发（BestAvailableRule）
  • 重试（RetryRule）

3. 声明式 HTTP 客户端（Feign / OpenFeign）

Feign 在 Ribbon 基础上封装，简化了服务间调用代码。

4. 熔断与降级（Hystrix / Sentinel）

Hystrix 工作原理：

  • 线程池隔离：每个服务独立线程池，防止雪崩
  • 信号量隔离：通过计数器控制并发
  • 熔断器状态机：CLOSED → OPEN → HALF_OPEN

服务熔断 vs 服务降级：

  • 熔断：防止雪崩，当错误率达到阈值时自动熔断
  • 降级：返回 fallback 兜底数据，保证系统可用

5. 网关（Gateway / Zuul）

Gateway 的作用：

  • 统一入口
  • 鉴权认证
  • 路由转发
  • 限流控制
  • 跨域处理
  • 日志记录

6. 配置中心（Spring Cloud Config / Nacos）

分布式配置管理，支持配置的动态刷新。配合 Spring Cloud Bus 可通过消息队列广播配置变更。

7. 服务链路追踪（Sleuth + Zipkin）

追踪一次请求经过的所有微服务，可视化调用链，方便排查性能瓶颈和故障。

四、面试题

1. 微服务分服务的原则是什么？

DDD（领域驱动设计）原则：按业务领域拆分，每个微服务对应一个限界上下文（Bounded Context）。

2. 为什么这样分？

  • 高内聚低耦合：每个服务专注自己的业务领域
  • 独立演进：各个服务可以独立开发和部署
  • 团队自治：每个服务可由独立团队维护

3. 微服务和 SSM 哪个好？区别？

对比 微服务 SSM（单体）
架构 分布式，多服务 单体应用
部署 独立部署 整体部署
扩展 按服务独立扩展 整体扩展
开发效率 初期慢，后期快 初期快，后期慢
运维成本 高（需容器编排、监控等） 低
适用场景 大型复杂系统 中小型系统

没有绝对的好坏，根据业务复杂度选择。刚开始建议单体，随着业务规模扩大逐步拆分为微服务。
